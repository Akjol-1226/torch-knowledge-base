## 为什么改

PRD v0.2 完成时遗留三个未决问题，均为开发前必须收口的事项：

- Q-001 不解 → 无法估入库成本与本地化路径
- Q-002 不解 → 无法搭建脚手架
- Q-003 不解 → 队列容量 / 索引规模 / VLM 预算都拍不下

用户在本次会话给出全部三个答案（云 VLM / 后端 Python / 数据 20 文档 × 40-130 页），同时把前端选型授权给我决定。需要立刻固化为正式决策，避免未来反复。

## 思路

1. **三个问题打包成一个 ADR**（而不是 ADR-001/-002/-003 各写一份）：三者强耦合（数据规模决定云模型可行性、语言栈决定 langchain 落点、VLM 路径决定 v0 vs 客户侧部署形态），分开写会重复论证。
2. **前端选型走候选对比**：列 Next.js / Streamlit / HTMX / Reflex 四方案，按 5 个维度打星（mockup 复用、流式、演示效果、招聘、单语言栈）。结论 Next.js 在前 4 项主导，只输「单语言栈」一项 — 而单语言栈代价（FastAPI ↔ Next.js 是成熟组合）远低于「扔掉 1587 行 mockup」与「丢失视觉精度」。
3. **关键决策约束写进 ADR**，不只是写选什么：
   - `VLMClient` 抽象层（云/本地热替换）
   - 厂商优先「云版与开源版同源」（如 Qwen），降低后期切换 prompt 差异
   - OpenAPI → TS client 自动生成（缓解双语言栈类型同步代价）
4. **遗留动作显式列出**：ADR-002（VLM 厂商）、ADR-003（BM25 后端）、客户对齐扫描件比例、抽样 PoC — 不藏在正文里，让 reviewer 一眼看到「还差什么」。
5. **数据规模评估代入数字**：20 × (40-130) = 800-2600 页；VLM 单价 $0.005-0.01/页 → $10-25 全量；明确「单机够」「不上 Celery」，避免过度工程。

## 备注（给下个 reviewer / 开发者）

1. **ADR-001 不锁定 VLM 具体厂商**，只锁定「走云路径 + 优先同源开源」。具体选 Claude / Doubao / Qwen / GPT 由 ADR-002 + PoC 决定。
2. **双语言栈是接受了运维代价的决策**：v2+ 客户内网部署时打包复杂度会上升，应预留时间。建议早期用 docker-compose 标准化，别走野路子。
3. **OpenAPI codegen 不是可选项**，是双语言栈类型同步的关键缓解。Day 1 就要接进 CI。
4. **`VLMClient` 抽象层**是切本地化的关键。所有入库代码不允许直接调云 SDK，必须经过这层抽象。
5. **数据规模可能膨胀**：v0 测试 20 文档是客户给的下限，v1+ 真实规模可能上千。PageIndex 树存储、入库流水线接口要从一开始就预留可扩展（不是现在就实现，是设计时心里有数）。
