## 为什么改

- 把 v0 brainstorm 阶段的所有产品决策固化为正式 PRD，作为开发与 review 的 SSOT
- 客户（火炬电子）的工艺/检验文档知识库问答系统需要明确的产品定义，避免开发期反复返工
- harness 规范要求重大产出走 product/prd/ 路径

## 思路

1. **采用对话式 brainstorm 收集需求**：调用 `superpowers:brainstorming` skill，遵循"一次一个聚焦问题 + 多选项 + 我推荐 + 风险提醒"的节奏，逐步把三大功能（入库/聊天/权限）的产品形态聊透
2. **关键决策点**全部经用户确认：
   - 入库引擎：纯 VLM（用户拒绝 MinerU 因为需要 `<notsure>` 机制）
   - 检索方案：PageIndex 树形检索（不走向量）
   - 权限模型：业务角色动态创建 + 功能角色（管理员/普通用户）正交
   - IA：TopBar 模块切换 + 模块内 SideBar 子页
3. **风险显式记录**：R-001（summary 漂移）/ R-002（VLM 表格）/ R-003（原始 tool call 演示）—— 不藏在 PRD 体内，单独成章
4. **章节编号** ：写完后发现跳号 bug（缺 3），用 sed 批量 renumber，保持 1-9 连续
5. **不引入 emoji**：用户明确反馈过，已写入 memory，全文用 ASCII box drawing 替代

## 备注（给下个 reviewer / 开发者）

1. **业务角色动态化是设计的关键决策点**：不要把"瓷片工程师/芯片工程师"写死成枚举，管理员要能在「系统 > 角色管理」里随业务变化随时新建/合并/废弃角色
2. **纯 VLM 路线接受了 R-002 风险**：表格质量风险靠人工抽检 SOP 兜底，开发期必须建立这个 SOP
3. **PageIndex 是产品差异化点**：销售向客户讲故事时强调"像工程师翻手册"，比向量检索易讲、抗投诉
4. **v0 不做兜底回答**：找不到就老实说没有，**严禁用 LLM 通用知识乱答**（火炬电子标准跟通用标准可能冲突）
5. **PRD 里指标阈值都是占位**（"待与客户对齐"），开发前必须跟客户对一遍
