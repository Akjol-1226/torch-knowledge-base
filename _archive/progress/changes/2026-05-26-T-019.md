## 为什么改

backend 骨架 review 时发现 4 个 blocker：

- **P0-1**：刚写的 AGENTS.md R-2/R-6 引用 `app.shared.contracts` 和 `app.shared.exceptions`，但实际 `app/shared/` 目录不存在。业务开发者 `from app.shared.exceptions import NotFoundError` 直接 ImportError → 下发任务即翻车
- **P0-2**：AGENTS.md R-3/R-4 要求 LLM 调用走依赖注入 + 走 vlm/client.py，但 `VLMClient` 没有 Depends provider 函数。业务开发者要么自己 `VLMClient(get_settings())`（破坏 R-3），要么自造 provider（不统一）
- **P1-1**：原 fixture 用 `db_mod._engine = db_engine` 改全局变量。业务开发者照抄就是反模式（隐藏耦合 + 不能并发跑 test）
- **P1-3**：`get_current_user` 定义了但 0 个 router 用它。业务开发者没有"auth 怎么用"的可抄模板

不修这 4 项就下发任务，等同于把陷阱直接交给业务开发者。

## 思路

1. **shared/exceptions.py 一次性把常见异常基类备齐**：业务开发者不需要每个模块自造异常；新增异常时继承本模块的基类。`http_status + code` 双属性方便后续 main.py 的全局 exception handler 统一映射到 HTTP 响应
2. **get_vlm_client 用 module-level 单例**：v0 阶段简单可靠。pyright `reportAttributeAccessIssue` 强制 test 用 `from app.core.vlm import client as vlm_client_mod` 而非 `from app.core import vlm; vlm.client._vlm_client = None`（已修复一次 error）
3. **dependency_overrides 是 FastAPI 官方教程的推荐模式**：`monkeypatch._engine` 是反模式（hidden global state）。换过来后 test 之间互相隔离 + 业务开发者抄的是正确写法
4. **chat/echo 加 user 是最小化示范**：不引入新 endpoint，只在已有 endpoint 加一个 Depends，破坏面最小。同步改 1 个 test
