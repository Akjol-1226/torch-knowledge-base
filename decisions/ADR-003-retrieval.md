## 背景

v0 检索锁定 PageIndex 树 + 纯 BM25（`bm25s`），曾明确"不走向量检索"。BM25 擅长精确词
（型号/图号/工序号，配合域词典），但对**概念性 / 换种说法**的查询召回弱——关键词对不上就漏。
实测语料（695 节点）也显示长尾节点多，纯关键词召回不足。故引入向量做语义召回、与 BM25 混合。

## 决策

### D-1：`search_nodes` 内置 BM25 + 向量的加权 RRF 融合（单一工具）

向量检索**不**作为独立工具暴露给 agent，而是融合进现有 `search_nodes` 内部。agent 工具面、
SSE、前端、引用链路全部不变。

- **不选**"多工具让 agent 自选 BM25/向量/混合"：模型选检索策略不稳、增 token 与延迟、维护多路径。
- 检索策略的选择用 RRF 确定性做对，不外包给 agent 临场判断。

### D-2：用加权 RRF（倒数排名融合），不用加权分数融合

`score(id) = w_bm25/(k+rank_bm25) + w_vec/(k+rank_vec)`，只对进了候选的那一路计分。

- 弱/稀释向量（如长节点单向量）排名靠后或缺席 → 对该节点"沉默"（0 贡献）而非负分，**不拖累 BM25**。
- 加权**分数**融合会让低向量分直接拉低混合分，正是要规避的，故不用。
- 默认 `w_bm25=0.6, w_vec=0.4, k=60`；另设**向量准入阈值** `vec_sim_threshold=0.30`，低相似度命中不参与融合。

### D-3：embedding 走 LiteLLM Proxy，默认 `text-embedding-v4`，本地 onnxruntime 兜底（预留）

- 复用团队统一入口与凭证；默认 `text-embedding-v4`（Qwen，1024 维，中文友好，对齐 Qwen docparse 栈）。
- `EmbeddingClient` 接口化，`provider=local` 预留本地 onnx 中文模型（本期未实现）。
- 向量是增强项、BM25 是底线：embedding 不可用 / 索引缺失 / `hybrid_enabled=False` / 向量空间签名不匹配
  → 一律优雅退回纯 BM25。

### D-4：单向量/节点 + 截断，不做长节点 chunking（YAGNI）

- 编码输入 = `title+summary+text`（summary 在前，截断只丢正文尾部），截断到模型上限（config，约 8K）。
- 91% 节点不长；长节点针尖问题 BM25 全文已兜住。评测确认漏召回再针对性给长节点加 chunking 多向量。
- 规模小（~700 向量），内存 numpy 点积，不引入 faiss/向量库。

## 后果

### 正面
- 概念性/换说法查询召回提升（实测每个样例查询的排序都被向量重排，命中更相关）。
- 对 agent / 前端 / 引用链路零侵入；返回结构逐字段不变。
- 全链路可优雅降级到纯 BM25，向量故障不影响可用性。

### 负面（已接受）
- 入库多一步 embedding（按内容 hash 缓存，未变节点不重编码）。
- 依赖 Proxy 的 embedding 模型可用性；切换模型需重建向量索引（签名机制保证不误用旧向量）。

### 衔接的后续动作（不在本 ADR）
- 用户反馈驱动的"检索学习"（把反馈变成排序信号），另开 spec，建在本层之上。
- 评测集 + 调参（权重/阈值/top_n）；必要时给长节点加 chunking。
