## ① 核心用户故事 + 主流程

### 1.1 业务背景

**客户**：火炬电子（Torch Electronics），从事芯片、多层瓷片电容（MLCC）、电容器等电子元器件研发与制造。

**痛点**：研发工程师遇到生产/质量问题时，需要翻共享盘里的工艺文件 / 检验标准 / 历史记录。文件多、流程长、细节藏在 200 页 PDF 的某一段；扫描件不可 Ctrl-F，跨文档关联靠经验。平均排查耗时 20-30 分钟，新员工尤其痛苦。

**v0 价值**：把工程师从"手翻 200 页"的体力活，变成 **"问一句 → AI 翻到对应章节 → 一键看原文"** 的脑力活。

### 1.2 用户角色

两个**正交独立**维度：

| 维度 | 取值 | 说明 |
|---|---|---|
| **业务角色** | 由管理员在「系统 > 角色管理」中**动态创建**（如：瓷片电容研发 / 芯片研发 / 质量检验 等） | 决定能进哪些知识库 |
| **功能角色** | 普通用户 / 管理员（系统内置，不可删） | 决定能做哪些操作 |

一个用户同时拥有 0 或多个业务角色 + 1 个功能角色。

> 业务角色动态化的理由：火炬电子真实部门会随项目变化，写死成枚举会让产品适应性差。

### 1.3 用户画像

**技术人员（普通用户）**
研发/工艺/检验业务团队成员。遇到生产问题需快速查标准、规范、历史案例。期望自然语言提问 + 看到答案出处 + 一键跳原文。

**管理员**
知识库负责人（IT / 项目经理 / 资深工程师）。批量上传历史文档、确认 VLM 不确定的识别结果、配置角色与权限、追溯改动。期望异步处理 + 任务状态可见 + 留痕完备。

### 1.4 核心用户故事

#### 故事 1：技术人员排查生产问题（最高频）

> **作为**瓷片电容工程师，
> **我希望**能用自然语言问"膜厚怎么控制"，
> **以便**在 2 分钟内拿到带原文引用的标准答案，而不是花 30 分钟翻 200 页扫描件。

**主流程**：
```
1. 进入「聊天」 → 输入"膜厚控制要求是多少？"
2. AI 显示 tool call 过程（默认折叠）—— PageIndex 检索：summary → 目录 → 章节
3. AI 输出："膜厚应控制在 50±5μm [1][2]，高温环境需考虑收缩率 [3]。"
4. 右侧引用面板列出 3 个来源章节
5. hover [1] → 右侧高亮；点击 [1] → 看 Markdown 原文
6. 想看原 PDF → 点 [PDF] → 新窗口跳转到 PDF 第 47 页
```

#### 故事 2：新员工快速上手

> **作为**新入职的技术人员，
> **我希望**不用打扰老员工就能找到相关标准文档，
> **以便**独立完成排查工作并加快 onboarding。

**主流程**：直接问 AI → AI 翻多份相关文档目录 → 定位具体条款 → 新员工直接读到，10 分钟内拿到答案（vs 原 1-2 小时）。

#### 故事 3：管理员入库新版本工艺文档

> **作为**管理员，
> **我希望**能批量上传几十份大文档而不卡死浏览器，
> **以便**异步处理完成后回来逐份 review VLM 识别的不确定项。

**主流程**：
```
1. 「知识库管理 > 上传」→ 选目标库 → 拖拽 V4.pdf → 提交
2. 任务进队列 → 关掉浏览器去做别的事
3. 30 分钟后回来查「任务队列」→ V4 状态 "待 review"
4. 点"去 review" → 进 review 详情页
5. 文档识别出 28 处 <notsure>：18 处标正确 / 6 处修改保存 / 4 处跳过
6. 处理完文档状态 → "已入库"
7. 把旧版 V3 标记"已废弃"（不删除，保留历史）
```

### 1.5 三大主流程图

```
[主流程 A：入库]
管理员上传 → 异步队列 → 纯VLM识别 → 段落级<notsure>标记 →
PageIndex建树+leaves索引 → "待review"状态 → 管理员 review →
"已入库"状态 → 可被聊天检索

[主流程 B：聊天]
用户提问 → LLM分析意图 → PageIndex双向检索：
   自上而下（summary→目录→章节）+ 自下而上（BM25→leaves→上溯）
→ LLM综合结果 → 输出答案+数字徽章引用 → 右侧panel展示来源

[主流程 C：权限]
管理员创建业务角色 → 关联知识库 → 分配给用户 →
用户提问时AI检索范围 = 该用户所有业务角色覆盖的知识库的并集
```

## ② 业务边界（什么不做）

### 2.1 v0 不做（v1 推迟评估）

- Tool call 步骤卡片美化（v0 用原始 JSON）
- 长会话 summary 压缩（v0 直接截断旧消息）
- 知识缺口飞轮（统计"无答案"反推文档补充）
- 操作日志页面（v0 仅写日志表不暴露 UI）
- 库级管理员（v0 全局管理员）
- 多语言（v0 仅中文）
- 图片直传提问（v0 仅文字）
- Excel / CAD 文件入库
- 知识库间引用关联
- 外部通知（邮件 / 钉钉 / 企微）
- 移动端原生 App（v0 响应式 Web 即可）
- 公司 SSO / LDAP 接入

### 2.2 永不做（v0/v1 都不做）

| 不做 | 理由 |
|---|---|
| 用 LLM 通用知识兜底回答 | 火炬电子标准可能与行业通用标准冲突，乱答会出事故；找不到就老实说没有 |
| 对外服务 | 仅限火炬电子内部 |
| 用户自主上传 | 仅管理员可上传，避免污染知识库 |
| 替代专家经验 | AI 只回答"文档里有的内容"，不做经验性推论 |

## ③ 验收标准（可量化）

### 3.1 业务指标（v0 上线 1 个月后达成）

> 阈值具体数值**待与客户对齐**，下表为指标定义。

| 指标 | 定义 | 验证方式 |
|---|---|---|
| 日均提问数 | 全用户每日提问总次数 | 后台埋点统计 |
| 平均回答带引用数 | 每个回答的引用数量平均值（>=1 说明 AI 真在"翻书"） | 后台埋点统计 |
| "无答案"响应占比 | 触发"未在知识库找到相关内容"的提问占比（过高说明文档覆盖不够） | 后台埋点统计 |
| 核心研发主动反馈"省了我的时间"人数 | 定性访谈 / 满意度调研 | 与客户共同执行 |

### 3.2 性能指标

| 项 | 阈值 | 备注 |
|---|---|---|
| 聊天首 token 延迟 | < 3 秒 | 流式输出 |
| 聊天完整回答 | < 30 秒 | 含 PageIndex 双向检索 |
| 入库 VLM 处理速度 | 单页 5-30 秒 | 受云模型限制 |
| 单文档入库 200 页 | < 1 小时 | 含 VLM + 建树 + 索引 |
| 并发用户上限 | v0 假设 < 100 | 待与客户对齐 |

### 3.3 安全合规指标

| 项 | 要求 |
|---|---|
| 传输加密 | HTTPS / TLS 1.2+ |
| 密码存储 | bcrypt / argon2 哈希，禁止明文 |
| 数据存储 | 文档原文件 + Markdown + 索引数据**全部存客户内网**（部署阶段决定） |
| 留痕保留 | 关键写操作 who/when 至少保留 3 年 |
| 敏感库隔离 | 查询时强制按用户业务角色过滤；越权访问返回 403 |
| 日志脱敏 | 错误日志不得包含完整文档内容；仅记录 doc_id + 错误码 |

### 3.4 功能完整性 checklist

**入库**：
- [ ] 支持 PDF / Word / PPT 上传，单文件 ≤ 200MB
- [ ] 支持拖拽批量上传，同批文件绑定同一目标库
- [ ] 任务异步执行，并发数可在 UI 配置
- [ ] 失败自动重试 3 次（指数退避 30s / 2min / 10min）
- [ ] 段落级 `<notsure>` 标记
- [ ] Review 界面：左原图 + 右识别（文本框直编辑）
- [ ] 三个动作：跳过 / 标记正确 / 修改并保存
- [ ] 任务状态机完整：排队 → 处理 → 待review → 已入库（任意阶段 → 失败）
- [ ] leaf 改动留痕：who / when / before / after / op_type
- [ ] leaf 修改后异步重生成受影响祖先节点 summary

**聊天**：
- [ ] PageIndex 双向检索（自上而下 + 自下而上）
- [ ] 多轮会话 + context 累积
- [ ] 会话列表：永久保留 / 可命名 / 可收藏 / 可删除
- [ ] 数字徽章 `[1]` `[2]` `[3]` 引用

**权限**：
- [ ] 业务角色管理员可动态创建 / 编辑 / 删除
- [ ] 业务角色关联多个知识库（多对多）
- [ ] 功能角色仅"管理员 / 普通用户"两种且内置不可删
- [ ] 用户管理：新建 / 编辑 / 禁用（保留账号不删除）
- [ ] 关键写操作全部留痕

**IA**：
- [ ] TopBar 含「聊天 / 知识库管理 / 系统」三个入口
- [ ] 普通用户仅看到「聊天」
- [ ] 管理员看到全部三个入口

## 附录 A：功能模块细节

### A.1 入库模块

#### A.1.1 输入规格

| 项 | 规格 |
|---|---|
| 支持格式 | PDF（扫描件 / 数字版）/ Word（.docx）/ PPT（.pptx） |
| 单文件大小 | 200 MB |
| 批量上传 | 同一批次绑定同一目标知识库 |
| 字符集 | 中文为主，兼容英文/数字/工程符号（μ Ω ± 等） |

#### A.1.2 处理管线

```
原始文件
   ↓
[纯 VLM 视觉识别] —— prompt 要求 VLM 用 <notsure>…</notsure> 标记不确定段落
   ↓
[输出统一 Markdown（含 <notsure> 标记）]
   ↓
[PageIndex 建树] —— 章节层级 + summary + leaves 全文索引（BM25）
   ↓
[落库] —— 文档元信息 + 树 + 索引 + 原 PDF 文件保留
   ↓
[任务状态：待 review]
```

#### A.1.3 任务状态机

```
[排队中] → [处理中] → [待 review] → [已入库]
   ↓          ↓             ↓
   └──────────┴─────────────┴────→ [失败]（自动重试 3 次后仍失败）
```

失败原因示例：VLM 接口超时 / 文档损坏 / API 余额不足 / 文件格式异常。失败任务进入失败列表，管理员可看错误详情、手动重跑。

#### A.1.4 异步队列规格

| 项 | 规格 |
|---|---|
| 并发数 | UI 可配（默认 3，最大值与 VLM 服务限速对齐） |
| 失败重试 | 自动 3 次，指数退避 30s / 2min / 10min |
| 进度感知 | 任务队列页面实时显示；v0 不主动外推通知 |
| 队列优先级 | FIFO，v0 不做手动置顶 |

#### A.1.5 Review 流程

入口：「任务队列」中状态 = 待 review 的行 → 点"去 review"；或「待 review」侧栏 → 点"开始 review"。

页面布局：左原图（黄框高亮 notsure 区） + 右识别结果（文本框直接编辑 + 三个动作按钮）+ 底部上下处导航。

完成所有 notsure 处理或主动退出：文档状态自动从「待 review」→「已入库」（保留未处理的 `<notsure>` 标记进库）。

#### A.1.6 改动留痕

每次 leaf 修改记录：
- `who`：操作者用户 ID
- `when`：时间戳
- `before`：原 Markdown 内容
- `after`：修改后 Markdown 内容
- `op_type`：跳过 / 标记正确 / 修改并保存 / 后期重新修改

入口：「知识库管理 > 知识库 > 进入某库 > 改动留痕」时间线视图。

#### A.1.7 重新入库（leaf 级）

事后管理员可定位到任意 leaf 修改 → 触发：
- 更新该 leaf 的全文索引
- 标记受影响祖先节点（章节 summary / 文档 summary）为"待重新生成"
- 后台异步重生成 summary
- 记录到改动留痕

### A.2 聊天模块

#### A.2.1 检索方式（PageIndex 双向）

```
用户问题 → LLM 分析意图 → 双路并行：

(A) 自上而下：summary → 目录 → 章节 → leaves（像工程师翻手册）
(B) 自下而上：BM25 匹配 leaves → 返回 [文档/章节/leaf 内容] 三元组 → LLM 判断相关性 → 反向追上下文

LLM 综合 (A)(B) → 输出答案 + 引用
```

> 技术细节由开发团队后续设计（langchain/langgraph 编排）。本节仅规定行为契约。

**销售亮点**：这种"像工程师翻手册"的检索体验是产品差异化点，比通用 RAG 易懂、可信、抗投诉。

#### A.2.2 Agent 透明度

答案下方一行可展开的 tool call 摘要 → 点击展开看原始 call/response JSON。v0 直接展示 JSON（工程师友好），v1 包装成步骤卡片（详见 R-003）。

#### A.2.3 引用展示

- 数字徽章 `[1]` `[2]`，hover 显示文档名 tooltip
- 右侧引用面板始终可见
- 颗粒度：每个引用 = 一个 leaf（章节/段落）
- 同文档多次引用 → 多张卡片分开
- 卡片含库标签（提示权限边界）
- 默认 Markdown，可切换原 PDF

#### A.2.4 多轮对话与会话管理

- 多轮 context 累积，超长截断旧消息（v0 不做 summary 压缩）
- 会话列表常驻左栏：自动生成标题 / 可手动改 / 可收藏 / 可删除 / 永久保留 / 跨设备同步
- 跨会话引用历史：v0 不支持

#### A.2.5 跨知识库检索

默认在用户**所有有权限的知识库**内自动检索。无 UI 选择器。多库混合引用时卡片显示库标签区分。

#### A.2.6 无答案处理

```
未在您有权限的知识库中找到相关内容。建议：
- 联系工艺部门或相关项目负责人确认
- 或检查是否需要管理员补充相关文档
```

禁止用通用知识兜底；禁止"似是而非"地拼凑。

### A.3 权限模块

#### A.3.1 业务角色（动态）

- 管理员在「系统 > 角色管理」创建
- 字段：名称 + 描述 + 关联知识库（多选）
- 用户拥有 0 或多个业务角色 → AI 检索范围 = 关联库的并集

#### A.3.2 功能角色（内置）

| 功能角色 | 能力 |
|---|---|
| 普通用户 | 仅聊天问答 |
| 管理员 | 全局：上传 / 改 notsure / 管用户 / 管知识库 / 管业务角色 |

#### A.3.3 用户管理

- 列表：用户名 / 业务角色集合 / 功能角色 / 可见库
- 操作：新建 / 编辑 / 禁用（不删除，保留审计）

#### A.3.4 登录方式（v0）

- 独立账号 + 密码（系统自有账号）
- 不接 SSO / LDAP（除非客户明确要求）
- 密码策略：>= 8 位含字母+数字

#### A.3.5 留痕（跨模块）

所有写操作记录审计：leaf 修改 / 用户管理 / 角色管理 / 知识库管理。v0 仅 leaf 改动暴露 UI（在文档详情页「改动留痕」tab）；其他写入日志表备查，v1 加操作日志页面。

## 附录 C：已知风险

### R-001：PageIndex 上层 summary 与 leaf 修改的一致性漂移

**场景**：管理员 review notsure 改了某个 leaf 内容，但该 leaf 所在章节的 summary、文档级 summary 仍基于旧内容生成 → 语义漂移。

**长期后果**：自上而下"翻书"检索找偏 → 工程师拿到错答案。

**缓解方案**：
- leaf 修改后自动标记受影响祖先节点为"待重新生成"
- 后台任务异步重生成 summary
- 定期跑一致性校验任务（v1）

### R-002：纯 VLM 处理复杂表格的结构性失败

**场景**：多列合并、跨页表格、嵌套表头时 VLM 容易行列错位**且自己感知不到**（不会标 `<notsure>`）。

**长期后果**：表格数据错误进入知识库，但不暴露为 notsure → 工程师无从识别。

**缓解方案**：
- 上线后**第一批入库的表格密集文档**必须人工抽检（10% 比例）
- 形成"VLM 表格抽检 SOP"作为入库质检流程
- v1 评估"重要表格用 OCR + VLM 双路对比" 的二次校验

### R-003：原始 tool call/response 在客户演示时显得粗糙

**场景**：v0 展示原始 JSON，给客户高层演示时非技术人员看了懵。

**长期后果**：产品体验观感差，可能影响验收。

**缓解方案**：
- v0 工程师内部用户友好，作为快速迭代起点
- v1 必须加折叠 + 步骤卡片美化层（参考 Claude.ai / Cursor）
- 客户演示前若 v1 未上，准备一段"专门讲清楚为什么这样设计"的话术

### Q-001：VLM 选型 — 已决（方向）

**问题**：v0 用云上模型起步，但要兼顾后期本地化迁移。具体选哪一个？

**候选**：
- 云：Claude Sonnet 4.6 Vision / GPT-4.1 Vision / Doubao 1.5 Vision / Qwen-VL-Max

**已决**（2026-05-26，ADR-001 D-1）：

- v0 走**云 VLM**路径，v1+ 评估本地化迁移
- 选型优先「云版与开源版同源」的厂商（如 Qwen），降低未来切换的 prompt 差异
- 入库 pipeline 抽象 `VLMClient` 接口，云/本地实现可热替换
- 具体厂商（Claude vs Doubao vs Qwen 云版）由 **ADR-002（待写）** 收口，需要先做抽样 PoC

**遗留动作**：抽样 5 份代表性文档跑端到端 PoC，对比识别准确率 / `<notsure>` 服从度 / 单 token 成本

### Q-002：前后端语言栈 — 已决

**问题**：langchain 生态在 Python 远比 TS 完整。前后端语言栈如何选？

**候选**：
- A. 全栈 TS（Next.js + LangChain.js）—— 一致性好，生态弱
- B. 前 Next.js (TS) + 后 Python (FastAPI + langchain/langgraph) —— 各取所长，运维复杂

**已决**（2026-05-26，ADR-001 D-2）：

- **方案 B**：前 Next.js 14 (App Router) + TypeScript + Tailwind + shadcn/ui；后 FastAPI + langchain/langgraph + pydantic v2
- 协议 REST + SSE，OpenAPI → TS client 自动生成
- 关键依据：mockup 已是 Tailwind / sibling 已是 Next.js / langchain Python 生态 / 演示精度要求

### Q-003：数据规模 — 部分已决

**问题**：客户实际有多少文档？哪种类型占比最高？扫描件 vs 数字版比例？

**对设计的影响**：
- VLM 入库总成本（关键预算项）
- 索引规模
- 并发与队列容量配置

**已决**（2026-05-26，ADR-001 D-3）：

- v0 测试规模：**20 文档 × 40-130 页 ≈ 800-2600 页**
- VLM 全量入库成本估 $10-25，可试错
- 队列方案：FastAPI BackgroundTasks / 单进程 asyncio queue（不上 Celery）
- BM25：SQLite FTS5 或 Postgres tsvector（ADR-003 收口）

**遗留动作**：与客户对齐扫描件 vs 数字版比例，影响 OCR 兜底策略
