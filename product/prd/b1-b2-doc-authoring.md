## FR-009 · B-1 基于历史框架生成新文档

### 1. 要解决的问题

工程师做新产品时，工艺/检验文件的**章节框架高度复用**——换个规格、改几个工序参数，其余照抄历史文档。现状是手动复制老文档、逐处改、还容易漏改联动值（如封装尺寸变了忘了改电容值）。

B-1 让工程师**选一份历史文档当蓝本**，用对话把"要改什么"说清楚，Agent 沿用其框架、只替换受影响的参数，产出新文档初稿。

**一句话边界**：B-1 是"换框架、批量参数替换、产出一份全新文档"；不是在原文档上改（那是 B-2）。

### 2. 核心交互流程（点击后理论上发生什么）

分 4 个阶段。每阶段写清：**用户动作 → Agent 做什么（技术逻辑）→ 界面展示什么**。

#### 阶段 1 · 选蓝本

| | 内容 |
|---|---|
| 用户动作 | 从知识库选一份历史文档作蓝本（未来支持产品系列 × 文档门类双维筛选 + 搜索；也可上传外部参考文档） |
| Agent 做什么 | 无 —— 仅记录选中的 `doc_id` |
| 界面展示 | 蓝本卡片列表（编号 / 规格 / 章节数 / 被引用次数 / 更新时间等"选模板信号"） |

#### 阶段 2 · Agent 通读蓝本 → 概括 + 反问

**关键原则：点开蓝本后，绝不立即列参数。** 一份 200 页文档可能有几百个参数，一次性甩出来是假交互。Agent 先"读一遍"，只概括、再反问。

| | 内容 |
|---|---|
| 用户动作 | 无（等待） |
| Agent 做什么 | 调 PageIndex **自上而下**读：`list_catalog` / `get_outline` 拿章节树 + **各 node 的 summary**（读摘要快速建立全局理解，不逐字读全文，省 token）。归纳文档骨架与适用对象 |
| 界面展示 | ① 一段**文档概括**（这是什么文档、锚定什么规格）；② **一句反问**："你想基于它做份什么新文档？" —— 直接从概括开始，不展示"正在读取/已通读"之类过程态，也不出现任何参数清单 |

技术依据：复用主 PRD A.2.1 的"自上而下翻书"检索，node summary 是 PageIndex 建树时已产出的现成物。

#### 阶段 3 · 用户给需求 → Agent 深读定位 + 计算 → 出改动清单

| | 内容 |
|---|---|
| 用户动作 | 用自然语言说需求（"做 1206 规格、涂布厚度改 15μm、型号跟着改，其余沿用；尺寸/电容你重算"） |
| Agent 做什么 | ① 解析需求，**定位受影响 node**（检索 + 按需 `read_node` 读这些节的**原文**，不是全文通读）；② **Agent 自主判断要改动哪些地方**（哪些 node、改成什么值），用户不逐一指派——系统只负责把每处改动的**来源**标注清楚：是"用户明确指定的"，还是"AI 推理决定要改的"；③ 对需要重算的值**调用计算 tool**（如规格 0805→1206 牵动物理尺寸、额定电容）——**只给 Agent 一个计算 tool、由它自主判断何时调用，不写死规则、不穷举公式**；④ 汇总成改动清单 |
| 界面展示 | **改动清单（plan）**，每条含：`位置`（第 X 章/附表）· `旧值 → 新值` · `来源标签`：**「你指定」**（用户明确要求）/ **「AI推理」**（Agent 推理判断需要改、需重点核对）· **改动原因备注**（每处都在前端展示：为什么改这里、依据是什么）。清单可勾选（取消勾选=不改）。底部「确认生成初稿 / 继续调整」 |

设计意图：仿 Claude Code 的 **plan 模式**——先给清单让人 review，confirm 后才动手，不一步到位。整份文档只列**受影响的**改动点，其余原样沿用不进清单。

#### 阶段 4 · 确认生成 → 展示 + 复制

| | 内容 |
|---|---|
| 用户动作 | 点「确认生成初稿」；可继续对话微调 |
| Agent 做什么 | 按蓝本框架 + 已确认改动，渲染新文档 |
| 界面展示 | 右栏新文档初稿：改动处**黄色高亮**；**AI 推理值用虚线下划线**提示务必核对；未变章节标"原样沿用"。顶部「初稿·待审核」徽章 + **复制按钮**（一键复制全文，粘回 Word）|

### 3. 输出与边界

| 项 | 规格 |
|---|---|
| 产出使用 | v0 只提供**复制按钮**（复制全文粘回 Word）。**导出 MD / JSON 先不做、不入库**（Q-B1-1 已决）；**不生成 .docx**（决策 D2）；后续粘贴回 Word 定位由 B-3 辅助 |
| 责任边界 | 初稿**仅供参考，必须经研发人员审核**。AI 推理值默认不可信、需人工核对实际工程值 |
| 计算 tool | 只提供能力、不写死业务规则；Agent 自主决定调不调、怎么算。避免过度设计 |

### 4. 待决问题（B-1）

- ~~Q-B1-1~~ **已决**：生成的新文档 v0 **仅复制粘走、不入库**（一键入库后续再评估）。
- Q-B1-2：计算 tool 的能力范围与可信度声明 —— 算错了如何暴露给用户、是否给出计算过程？
- Q-B1-3：蓝本筛选的分类维度（产品系列 / 文档门类）取值需按火炬真实主营确定。

## FR-010 · B-2 现有文档模块级编辑

### 1. 定位（重要：不是新页面）

B-2 **不新增页面**，就是主 PRD 里"知识库 > 查看结构"那个**文档结构弹窗**（PageIndex 章节树 + 点叶子看正文）——在**叶子正文区多加两个按钮**，把只读升级成可就地编辑。

**一句话边界**：B-2 是"在正式文档上，选一个最小块、只改这块、留版本可回退"；不是换框架生成新文档（那是 B-1），也不是入库前的 notsure review（那是主 PRD A.1.5）。

**权限（Q-B2-2 已决）**：正式文档的模块级编辑**仅管理员**可操作（与主 PRD A.3.2 管理员权限一致），普通用户只读结构与正文。管理员改动**直接生效**，v0 **不做二人复核**——靠"改动留痕 + 审计 + 可回退"兜底。

### 2. 交互（叶子正文区新增什么）

在叶子（leaf）正文上方新增两个按钮 **「编辑」** + **「AI编辑」**（叶子本身**不带版本号**——版本是文档级的，见第 4 节）。以下几态就地切换，全程在弹窗内、不跳页：

| 态 | 触发 | 行为 |
|---|---|---|
| 只读（默认） | 点开叶子 | 正文 + 「编辑」「AI编辑」两个按钮 |
| 编辑 | 点「编辑」 | 正文变文本框，手动改，「保存 / 取消」 |
| AI编辑 | 点「AI编辑」 | 出现指令输入框，自然语言说怎么改 → 发送 |
| diff·待确认 | AI 返回 | 正文原地显示**红删绿增 diff**，「接受 / 重新说 / 放弃」 |

**约束：AI 只改最小可动单元（叶子块）。** 章节容器节点点开只展开、**不提供「AI编辑」**，并提示"AI 编辑请选下面的叶子块"。理由：范围越小越好核对、越不容易误改联动内容（与 B-1 的"逐条改动清单"同一设计哲学）。

### 3. 编辑之后展示什么

| 时机 | 展示 |
|---|---|
| 接受/保存后 | ① 该块正文**即时更新**；② **文档版本** `vN → vN+1`（版本是**文档级**的，不是单个块各自一套）；③ 结构树上被改过的叶子标「改过」；④ 这次改动追加到**文档修改历史**时间线 |
| diff 阶段 | 一行"AI 已读本块原文、按【指令】只改了这块"，下方红删绿增，附计算/改动说明 |

### 4. 修改记录在哪看（复用现有机制，不另造）

B-2 的"修改记录/版本"**直接挂在主 PRD 已有的 A.1.6 改动留痕**之上，不新建一套：

| 机制 | 来源 | B-2 的增量 |
|---|---|---|
| 改动记录字段 | A.1.6：`who / when / before / after / op_type` | `op_type` 增加枚举：`手动编辑` / `AI编辑` / `回退` |
| 编辑后索引/summary | A.1.7 重新入库（leaf 级） | 编辑 leaf 后**自动**：更新该 leaf 全文索引 → 标记祖先章节/文档 summary 为"待重生成" → 后台异步重生成 |
| summary 漂移防护 | R-001 | B-2 是 leaf 改动的主要来源之一，必须触发上面这条，否则"自上而下翻书"会找偏 |
| 全局审计 | A.3.5 留痕 | 所有编辑/回退写审计 |

**修改记录是"整份文档一条时间线"，不是每个叶子各自的版本号**：

- **粒度**：版本挂在**文档**上，全局递增。任意块的编辑 / AI编辑都追加到这一条时间线；叶子本身不带版本号，只在被改过时标「改过」。
- **入口**：结构弹窗顶部「修改历史」，或 知识库 > 文档详情 > 「改动留痕」（A.1.6 已有）—— 二者是**同一份数据**的两个入口。
- **每条记录**：列表只显示 `改动章节`（可多个）· `日期` · `改动人`；「改了什么」点「查看 diff」展开，另可「回退」。
- **回退**：回退某条改动，回退动作本身也追加为新的一条记录（审计链只增不改，见 Q-B2-1）。

### 5. 待决问题（B-2）

- Q-B2-1：**回退是否算一次新版本**（回退 v1 → 生成 v3=回退自v1）？倾向"是"，保证审计链只增不改、可追溯。**建议采纳。**
- ~~Q-B2-2~~ **已决**：正式文档编辑**仅管理员**可操作（与 A.3.2 一致）；管理员改动直接生效，v0 **不做二人复核**，靠改动留痕 + 审计 + 可回退兜底。
- Q-B2-3：一个块内**多处改动能否分别接受/放弃**，还是整块要么全接受要么全放弃（v0 先做整块级）。
- Q-B2-4：编辑与"重新入库 summary 重生成"之间的**一致性窗口**——重生成完成前，检索命中的是旧 summary，是否需要状态提示。

## 附：B-1 与 B-2 的关系

| | B-1（FR-009） | B-2（FR-010） |
|---|---|---|
| 对象 | 历史文档当**框架**，产出**新文档** | **同一份正式文档**本身 |
| 改动范围 | 全局、批量参数替换 | 单个最小叶子块 |
| 交互形态 | 独立对话页 + 全局改动清单 plan | 结构弹窗内、叶子正文就地 diff |
| 产物 | 新文档初稿（复制粘回 Word） | 原文档新版本（入库、可回退） |
| 审核 | 研发人员审核初稿 | 改动留痕 + 版本回退 + 审计 |
| 潜在衔接 | B-1 初稿入库后 → 即成为 B-2 可编辑对象 | — |
